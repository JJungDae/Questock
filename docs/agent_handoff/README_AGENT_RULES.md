> 목적: 로컬 개발 에이전트가 `PROJECT_PLAN_FINAL_PASS.md`를 실행할 때 적용할 안전·승인·검증·기록 원칙을 정의한다.  
> 적용 범위: 코드, 설정, 테스트, 문서, 데이터, Git, Docker, CI, 외부 API, 배포 관련 모든 작업

예외가 필요하면 사용자가 해제할 원칙과 수행할 명령을 구체적으로 승인해야 한다.

# 2. 핵심 안전 원칙

## 2.1 최소 수정 원칙

- 현재 승인된 작업의 완료 기준에 필요한 파일만 수정한다.
- 기존 구조를 전면 재설계하거나 대규모 리팩터링하지 않는다.
- 관련 없는 formatting, 이름 변경, 폴더 이동, dependency 추가를 함께 수행하지 않는다.
- 기존 코드로 해결할 수 있으면 새 abstraction, framework, service를 만들지 않는다.
- 수정 예상 파일과 수정하지 않을 영역을 작업 전 계획에 명시한다.
- 승인 범위를 넘는 변경이 필요하면 작업을 중단하고 최소 수정안과 대안을 보고한다.

## 2.2 확인하지 않은 대상을 추측하지 않는다

- 실제로 열어보지 않은 파일의 내용이나 존재를 가정하지 않는다.
- 확인하지 않은 함수, class, API, DB table, 환경변수, branch, workflow를 존재한다고 말하지 않는다.
- 이전 프로젝트나 유사 저장소의 구조를 현재 저장소에 그대로 적용하지 않는다.
- 문서와 코드가 충돌하면 실제 상태를 확인하고 차이를 보고한다.
- 확인하지 못한 사항은 `미확인` 또는 `추가 확인 필요`로 표기한다.

## 2.3 실행하지 않은 결과를 성공으로 주장하지 않는다

- 실행하지 않은 명령은 실행했다고 말하지 않는다.
- 테스트를 실행하지 않았다면 `NOT_RUN`으로 기록한다.
- 일부만 실행했다면 `PARTIAL`로 기록하고 제외 범위를 명시한다.
- Docker, CI, 외부 API, 배포를 실제 확인하지 않았다면 성공했다고 주장하지 않는다.
- 로그, exit code, 응답값 등 검증 증거가 없는 성공 주장을 금지한다.
- mock·fixture 성공을 live provider 성공으로 표현하지 않는다.
- 로컬 성공을 배포 환경 성공으로 표현하지 않는다.

## 2.4 승인 범위 밖 기능을 추가하지 않는다

- 현재 승인된 Task와 계획서에 없는 기능을 임의로 구현하지 않는다.
- “나중에 필요할 것 같다”는 이유로 schema, route, UI, dependency를 선행 추가하지 않는다.
- P0 완료 전 P1·M5·stretch 기능을 시작하지 않는다.
- 로그인, dense retrieval, streaming, Langfuse, 지원 종목 확대는 계획의 gate와 별도 승인을 모두 충족해야 한다.
- 새로운 아이디어는 구현하지 않고 `제안 사항`으로만 보고한다.

## 2.5 기존 사용자 작업과 동작을 보존한다

- 초기 `git status`에서 발견한 기존 수정·미추적 파일을 수정·삭제하지 않는다.
- 사용자 승인 없이 `stash`, `reset`, `restore`, `checkout`, `clean`, 파일 삭제·덮어쓰기를 실행하지 않는다.
- 현재 작업과 기존 변경의 충돌 가능성을 먼저 보고한다.
- 수정 전 관련 테스트와 현재 동작을 가능한 범위에서 확인한다.
- 변경 후 같은 테스트를 다시 실행해 회귀 여부를 확인한다.
- Critical set, wrong-company 차단, fake locator 차단, 투자 행동 권고 차단을 항상 보존한다.

## 2.6 증거 중심으로 보고한다

모든 결과 보고에는 다음을 포함한다.

- 변경 파일
- 변경 함수·class·설정
- 실행 명령
- 실제 결과
- 통과·실패·미실행 테스트
- 남은 위험과 제한
- rollback 제안
- 다음 Step 진입 가능 여부

# 3. 필수 작업 루프

모든 작업은 다음 순서를 따른다.

```text
저장소·문서 확인
→ 작업 전 계획
→ 계획 승인
→ 승인된 코드 수정·테스트
→ 결과 보고
→ 사용자 결과 확인
→ Git 작업 별도 승인
→ 승인된 Git 작업
→ 회귀 확인
→ 오늘자 작업 로그 기록
→ 다음 Step 계획
```

## 3.1 저장소와 문서 확인

작업 시작 전 다음을 실제로 확인한다.

- 최우선 실행 문서 2개
- 현재 Task와 연관된 참고 문서 절
- 실제 디렉터리 구조
- 현재 Git branch·working tree·최근 commit
- 기존 README·dependency·실행 진입점·테스트 구조
- 기존 작업 로그
- 현재 Step의 선행 조건
- 관련 코드·설정·테스트

이 단계에서는 읽기 전용 확인만 한다.

## 3.2 작업 전 계획

구현 전에 다음 형식으로 계획을 제출한다.

```markdown
## 작업 전 계획

- 현재 Task bundle / Step:
- 작업 목적:
- 실제 확인한 저장소 상태:
- 미확인 사항:
- 선행 조건 충족 여부:
- 수정 예상 파일:
- 수정하지 않을 영역:
- 구현 순서:
- 검증 명령:
- 예상 테스트:
- 관련 위험 ID:
- fallback / rollback 제안:
- 오늘자 작업 로그 Step ID:
- 사용자 결정이 필요한 항목:
```

첫 작업에서는 사용자가 승인한 이 계획을 **임시 Task Card**로 간주한다.  
승인된 내용을 `docs/TASK_CARDS/<STEP_ID>.md`에 저장하는 작업은 별도 승인 후 수행한다.

## 3.3 계획 승인

계획 승인에는 다음만 포함한다.

- 필요한 branch의 생성·전환
- 승인된 파일의 코드·설정·테스트 수정
- dependency 설치가 계획에 명시된 경우 해당 설치
- 테스트·smoke test 실행
- `git diff` 검토

계획 승인만으로 다음은 허용되지 않는다.

- commit
- push
- Pull Request 생성
- merge
- 배포
- destructive Git·DB 작업

계획 승인 표현 예:

- “진행해줘”
- “이 계획대로 작업해”
- “계획 승인”

## 3.4 승인된 범위 구현

- 승인된 파일과 기능만 수정한다.
- 미승인 contract·schema 변경이 필요하면 즉시 중단한다.
- 관련 없는 다수 파일 수정, destructive 작업, secret·권한 문제가 발견되면 중단한다.
- 중단 보고에는 문제, 확인 증거, 최소 수정안, 대안, 일정·테스트 영향을 포함한다.

## 3.5 실제 실행과 검증

검증 순서:

```text
targeted unit test
→ 관련 integration test
→ Critical set 관련 테스트
→ phase-appropriate smoke test
→ git diff 검토
```

상태:

- `PASS`: 실제 실행했고 통과
- `FAIL`: 실제 실행했고 실패
- `PARTIAL`: 일부 실행
- `NOT_RUN`: 실행하지 않음
- `BLOCKED`: 환경·권한·외부 의존성으로 실행 불가

## 3.6 작업 결과 보고

```markdown
## 작업 결과 보고

- Task / Step:
- 변경 파일:
- 핵심 변경:
- 실행 명령:
- 테스트 결과:
- 실행하지 못한 검증:
- 현재 제한:
- 관련 위험:
- rollback 제안:
- 다음 Step 진입 가능 여부:
- 사용자 확인이 필요한 내용:
```

이 결과 보고는 해당 Step의 **임시 HANDOFF** 역할을 한다.  
사용자 확인 후 저장소에 정식 HANDOFF 문서를 작성할지는 별도 승인을 받아 결정한다.

## 3.7 사용자 결과 확인

다음 표현은 결과 확인으로 본다.

- “확인했어”
- “이 결과로 완료 처리해”
- “작업 로그에 기록해”
- “결과 승인”

다음은 결과 확인으로 보지 않는다.

- 단순 질문
- 추가 설명 요청
- 수정 요청
- 오류 지적
- “계속해”처럼 완료 여부가 불명확한 표현

“commit 진행해”는 결과 확인과 해당 commit 승인으로 볼 수 있지만, push·PR·merge·배포 승인은 포함하지 않는다.

## 3.8 Git 작업 별도 승인

결과 확인 후에도 다음 작업은 각각 승인 범위에 포함된 경우에만 수행한다.

1. commit
2. push
3. Pull Request 생성
4. merge
5. 배포

예:

- “commit만 진행해”
- “commit하고 push해”
- “PR까지 만들어”
- “merge해”
- “배포 진행해”

승인되지 않은 후속 Git 작업은 자동으로 수행하지 않는다.

## 3.9 회귀 확인과 오늘자 작업 기록

Git 작업 이후 또는 사용자가 Git 작업 없이 완료 처리를 지시한 이후:

- 관련 회귀 테스트 결과 확인
- 승인된 내용만 작업 로그에 기록
- 실패·미실행 항목도 그대로 기록
- 다음 Step은 자동 구현하지 않음

기본 경로:

```text
docs/work_logs/WORK_LOG_YYYY-MM-DD.md
```

기존 저장소 규칙이 있으면 기존 경로와 형식을 우선한다.

## 3.10 다음 Step

작업 로그 기록 후 다음 Step의 계획만 제출한다.

```text
계획 → 승인 → 작업 → 결과 확인 → Git 승인 → 기록
```

루프를 새로 시작한다.

# 4. Git·파일 안전 원칙

## 4.1 초기 dirty working tree 처리

기존 수정·미추적 파일이 있으면:

- 해당 파일을 수정·삭제하지 않는다.
- handoff 입력 파일과 기존 사용자 변경을 구분하여 보고한다.
- `stash`, `reset`, `restore`, `checkout`, `clean`을 실행하지 않는다.
- 현재 Task와 충돌 가능성을 보고한다.
- 안전한 branch·작업 범위를 제안하고 승인을 기다린다.

## 4.2 handoff 파일 보호

다음 파일은 사용자 제공 입력이며 승인 없이 수정·이동·삭제·commit하지 않는다.

- `LOCAL_AGENT_HANDOFF_README.md`
- `LOCAL_AGENT_START_PROMPT.md`
- `README_AGENT_RULES.md`
- `PROJECT_PLAN_FINAL_PASS.md`
- `WORK_LOG_TEMPLATE.md`
- `TASK_CARD_TEMPLATE.md`
- `HANDOFF_TEMPLATE.md`
- 정합성·위험·평가 참고 문서
- 전달용 ZIP 파일이 저장소에 존재하는 경우 해당 ZIP

초기 `git status`에서 이 파일들이 untracked로 표시되면 기존 사용자 변경과 분리해 보고한다.

## 4.3 branch와 destructive 작업

- main에서 직접 개발하지 않는다. 사용자가 구체적으로 허용한 경우만 예외다.
- 한 branch는 하나의 명확한 목적을 가진다.
- Task bundle은 세션 계획 단위이며 branch 단위가 아니다.
- 한 bundle 안의 독립 구현은 최대 1~2개 branch·PR로 나눈다.
- 사용자 승인 없이 commit·push·PR·merge·rebase·reset을 실행하지 않는다.
- 사용자 승인 없이 파일 삭제·덮어쓰기·DB 초기화·migration rollback을 실행하지 않는다.
- secret, `.env`, credential, 개인정보를 commit하지 않는다.
- unrelated diff가 있으면 원인을 확인하고 현재 작업에서 제외한다.

# 5. 금융 RAG 안전 원칙

- 종목 식별 전에 provider를 호출하지 않는다.
- 다른 종목 문서가 섞이면 답변을 생성하지 않는다.
- provider timeout과 no-data를 구분한다.
- 검색 결과가 있다는 이유만으로 `complete`로 판정하지 않는다.
- Evidence에는 snippet과 URL 또는 안전한 locator가 있어야 한다.
- 로컬 절대 경로를 사용자 응답에 노출하지 않는다.
- 근거에 없는 숫자·날짜·단위를 생성하지 않는다.
- fixture 성공을 live source 성공으로 주장하지 않는다.
- 사용자에게 특정 증권의 매수·매도·보유를 권고·지시하지 않는다.
- 공시·뉴스 속 거래 사실은 중립적으로 요약할 수 있다.
- 목표가·손절가·익절가·확정 가격 예측·수익 보장을 생성하지 않는다.

# 6. 완료 정의

다음 조건을 모두 만족해야 완료다.

- 승인된 범위만 수정
- 변경 파일과 목적 일치
- 관련 검증을 실제 실행하거나 미실행 사유 기록
- 회귀 여부 확인
- 실행하지 않은 성공 주장 없음
- 범위 밖 기능 추가 없음
- 결과 보고
- 사용자 결과 확인
- 필요한 Git 작업의 별도 승인과 수행
- 오늘자 작업 로그 기록
- 다음 Step 자동 진행 안 함

# 7. 금지 사항

- 저장소 확인 없이 architecture 생성
- 계획 승인 전 구현
- 사용자 결과 확인 전 완료 기록
- 테스트 삭제·완화로 통과 처리
- 실패 기능을 성공으로 문서화
- 임의 dependency·framework 도입
